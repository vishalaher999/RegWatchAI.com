finding. The expanded view shows the full evidence trail so Sarah can verify
the AI's reasoning before approving — never just a title to rubber-stamp.

## v1 scope and gaps

- **Approve**: the only action wired conceptually in v1. Per the Progressive
  Autonomy Roadmap (Stage 2), approval is still required for every
  AI-generated task — there is no auto-promotion to "In Progress" yet.
- **Edit / Dismiss**: shown for completeness but not built in v1. A dismissed
  task should still leave an `AuditLog` row (override), per CLAUDE.md's "every
  AI decision logs ... " — not designed yet.
- **Evidence Trail** is the most important element of the card: it is what
  lets Sarah judge whether the AI's HIGH-impact classification (F3) and task
  draft (F4) are trustworthy for THIS specific pair, not just "trust the
  system" in general — consistent with SR 11-7's emphasis on explainability
  over blind reliance.
- **Owner field** is always "Sarah" in v1 (documented limitation, see
  `notes/Day-31-F4.md`) — no UI for reassigning owners yet.
- **Due date styling** (e.g. red if overdue) not designed yet — out of scope
  for v1.

## How this connects to the rest of RegWatch

- Cards are populated from `data/f4_tasks/tasks.json` / the `Task` table
  (`src/models.py`), one row per F3 HIGH finding the agent was run on.
- The "Generated by" / "Audit log" fields surface `AuditLog` rows written by
  `src/f4_tasks/agent.py` (`action=AuditAction.TASK_CREATE`) — this is F5's
  audit trail, made visible at the point where Sarah needs it most.
- Future stages (per `docs/Progressive-Autonomy-Roadmap-v1.md`) would extend
  this board with bulk-approve for high-precision classes and a queue view
  for LOW/MEDIUM findings that still require full manual review.
